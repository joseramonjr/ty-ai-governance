## Purpose

This document defines what it means for a
TY AI OS installation to be compliant. It
defines what qualifies an installation to
use the TY AI OS name and what disqualifies
it. It defines the current self-attestation
process and the path to future technical
verification.

This document applies to all external TY AI OS
installations across all four governance paths.

---

## Section I -- The Compliance Principle

One sentence defines TY AI OS compliance:

  A system that modifies the core doctrine
  is not TY AI OS.

This sentence was established in Part 118
(2026-04-01) and is the permanent foundation
of this standard. Everything else in this
document is an elaboration of that sentence.

The inverse is equally true: a system that
runs the unmodified core doctrine, maintains
an honest governance record, and operates
within its path requirements is a compliant
TY AI OS installation regardless of its size,
its location, its operator, or its use case.

Compliance is not about capability. It is
about integrity. A small installation with
one governed application that runs honestly
is compliant. A large installation that
modifies one governance invariant is not.

## Section II -- What Compliance Requires

A compliant TY AI OS installation must satisfy
all of the following:

### II.A -- Core Doctrine Integrity

  1. The installation runs the unmodified
     TY AI OS core doctrine as published at
     TYOVA (tyova.ai) and in the ty-ai-governance
     repository

  2. No governance invariant has been disabled,
     weakened, or circumvented

  3. No governance mechanism in TY AI OS, Jaya
     Runtime, Jayme AI, or Luke AI has been
     modified to expand what any component
     can do beyond core doctrine limits

  4. The seven permanent principles from
     00_MASTER_CHARTER.md are enforced without
     exception

  5. All non-weaponization prohibitions from
     00_MASTER_CHARTER.md are in effect

### II.B -- Governance Ledger Integrity

  1. The append-only governance ledger is
     present and unmodified

  2. No entries have been deleted, altered,
     or reordered after recording

  3. The ledger accurately reflects the
     installation's governance history

### II.C -- Path Requirements

  Path 1 (Solo, No Guardian):
    -- Agreement to core doctrine confirmed
    -- No additional requirements

  Path 2 (Independent, Own Guardian):
    -- Signed Continuity Charter on file
    -- Guardian designated in governance ledger
    -- Human Verification Protocol initialized

  Path 3 (Federated, Own Guardian):
    -- All Path 2 requirements
    -- Signed Federation Membership Protocol
    -- Registered in federation registry

  Path 4 (Federated, Shared Guardian):
    -- Signed Continuity Charter on file
    -- Signed Federation Membership Protocol
    -- Registered in federation registry
       with shared guardian reference

### II.D -- Operational Integrity

  1. The installation is used only for purposes
     consistent with the Master Charter mission

  2. The installation is not being used for
     any prohibited purpose listed in
     00_MASTER_CHARTER.md Section V

  3. The installation accurately represents
     itself -- it does not claim compliance
     it has not attained

## Section III -- What Disqualifies an
##                Installation

An installation is disqualified from using
the TY AI OS name and from federation
membership if any of the following are true:

  1. Any governance invariant has been modified,
     disabled, or circumvented

  2. The append-only ledger has been altered,
     deleted entries, or reordered entries

  3. The installation is being used for a
     prohibited purpose from
     00_MASTER_CHARTER.md Section V

  4. The installation falsely represents itself
     as compliant

  5. The installation has expanded what TY,
     Jaya Runtime, Jayme AI, or Luke AI can
     do beyond core doctrine limits

  6. For federation members: the installation
     has not fulfilled its federation membership
     obligations

Disqualification is not a penalty. It is a
factual description. A disqualified installation
may continue to operate as a non-TY-AI-OS system.
It simply may not use the TY AI OS name.

Disqualification can be reversed if the
installation is brought back into full
compliance and a compliant governance record
is re-established from the point of compliance
forward. Prior disqualified operation cannot
be retroactively corrected -- only new compliant
operation going forward is recorded.

## Section IV -- The Name TY AI OS

The TY AI OS name belongs to the compliant
unmodified system. It is not a brand that
can be licensed or applied to modified
variants.

An installation may call itself TY AI OS if
and only if it is compliant per this standard.

An installation may not call itself TY AI OS
if it has modified the core doctrine, disabled
any governance mechanism, or otherwise failed
the compliance requirements in Section II.

An installation may build on TY AI OS ideas,
publish documentation about TY AI OS, or
reference TY AI OS in academic or research
work. These activities do not require
compliance certification. They simply must
be accurate.

An installation that runs a modified version
of TY AI OS may call itself a fork or a
derivative. It may not call itself TY AI OS.

## Section V -- Self-Attestation

Compliance is currently self-attested. There
is no central authority that inspects
installations and grants compliance status.

Self-attestation means: the operator of the
installation declares under their own authority
that their installation meets all compliance
requirements in Section II.

Self-attestation is recorded in the
installation's governance ledger as a compliance
declaration entry.

For federation members, compliance status is
also visible in the federation registry.

Self-attestation is not a formality. It is
a genuine commitment. A false attestation
is a violation of the core doctrine principle
of Long-Term Institutional Memory -- the
requirement that governance history be
accurate and immutable.

A self-attestation declaration is provided
in Section VII of this document.

---

## Section VI -- Future Technical Verification

Self-attestation is the current compliance
mechanism. A technical verification mechanism
is planned for a future governed phase
(Phase 11 or later).

The technical verification mechanism will
allow a Jaya Runtime installation to
cryptographically verify that its core
doctrine files match the canonical published
versions at TYOVA. This verification will
supplement -- not replace -- self-attestation.

When the technical verification mechanism
is available it will be defined in a new
document and referenced here. Until that
time self-attestation per Section V is the
complete compliance mechanism.
